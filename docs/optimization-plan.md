# Post-Stage-11 Optimization Plan

## 背景

当前仓库已经完成 `Dioxus 0.7.3 + Axum 0.8.8` 的阶段 11 重写收口，核心能力已经可用：

- 管理台登录、登出、账户更新
- 订阅组创建、删除、排序、链接编辑
- 公共聚合路由 `GET /{username}`
- 持久 session、CSRF、防 SSRF、基础限流、持久 cache、抓取诊断

但从源码现状看，仍有一批值得优先处理的优化项，主要集中在：

- 认证正确性与安全边界
- 数据一致性与运行时内存边界
- 前端可观测性与可用性
- 测试、运维可见性与文档同步

本文件用于记录后续优化计划，不代表这些改动已经完成。

## 优先级判断原则

优化顺序按以下原则排序：

1. 先处理会直接影响安全、认证正确性、运行时稳定性的事项。
2. 再处理会导致脏数据、资源泄漏或用户误判的事项。
3. 然后补齐后端已有但前端未暴露的能力，提升实际可用性。
4. 最后补测试、观测、文档和较低风险的结构性清理。

## 优化清单

### P0: 认证与安全正确性

#### 1. 密码不应被 `trim()`

当前认证和改密路径会对密码做 `trim()`，这会改变真实密码语义。

风险：

- 前后空格会被静默忽略。
- 用户实际输入和后端验证规则不一致。
- 认证逻辑存在隐性兼容行为，后续很难安全演进。

目标：

- 用户名继续允许 `trim()` 后校验。
- 密码始终按原始输入参与校验、比对和哈希。

#### 2. 登录成功后轮换 session 标识与 CSRF

当前登录是在现有 session 上直接写入 `user_id`。

风险：

- 匿名态与登录态共用同一个 session 身份。
- CSRF token 也随旧 session 继续复用。
- session fixation 风险边界仍然偏松。

目标：

- 登录成功后轮换 session 标识。
- 登录、登出、账户更新后重置或重建 CSRF token。
- 保持当前前端自动重取 CSRF 的交互方式不变。

#### 3. 启动期配置硬校验

当前多个配置项允许非法或无意义的值进入运行态。

典型项：

- `CONCURRENT_LIMIT`
- `DB_MAX_CONNECTIONS`
- `FETCH_TIMEOUT_SECS`
- `MAX_LINKS_PER_USER`
- `MAX_USERS`

风险：

- 服务在坏配置下启动成功，但行为失真。
- 问题会拖到运行期才暴露，排查成本更高。

目标：

- 为关键配置增加显式校验。
- 非法值在启动阶段直接失败。
- 保留安全默认值，但不容忍无效覆盖值。

#### 4. 429 响应补齐标准头

当前限流只在错误消息中返回重试秒数。

目标：

- 为登录限流和公共路由限流补 `Retry-After`。
- 让客户端、代理和监控系统能识别限流行为。

### P1: 数据一致性与运行时边界

#### 5. 用户写路径改为事务化

当前用户删除、链接更新后，会在主写操作之后继续做 cache 和 diagnostics 清理。

风险：

- 某个后续步骤失败时，会留下部分成功、部分失败的状态。
- 逻辑一致性依赖应用层顺序，而不是数据库约束。

目标：

- 删除用户、更新链接等跨表写操作放入事务。
- 减少中间失败导致的残留状态。

#### 6. 补数据库级约束与级联清理

当前 schema 对 `users`、`fetch_diagnostics`、`user_cache_snapshots` 的关系约束仍偏弱。

目标：

- 补外键与合适的级联删除策略。
- 让“删除用户即清理诊断和缓存”成为数据库层保证，而不是只依赖应用代码。

#### 7. DNS cache 与 pinned client pool 增加容量边界

当前运行时缓存只有 TTL 或命中时清理，没有明确容量上限。

风险：

- 访问大量不同 host 时，内存持续增长。
- 长时间运行后，缓存对象规模不可控。

目标：

- 为 DNS 解析缓存引入有界回收。
- 为 pinned client pool 引入容量限制和淘汰策略。
- 保留现有“校验地址后固定连接目标”的 SSRF 防护思路。

#### 8. 评估链接安全校验的受控并行

当前保存链接时会对每个链接逐个做 `validate_safe_url(...)`。

影响：

- 当用户一次性保存很多链接时，等待时间会线性增长。

目标：

- 在不放松 SSRF 校验的前提下，评估受控并行校验。
- 并发度仍受全局或局部上限控制。

### P2: 前端可见性与用户体验

#### 9. 资源状态显式区分 loading / success / error

当前前端资源快照对错误基本采用“看起来像空数据”的处理方式。

风险：

- UI 难以区分“接口失败”和“真的没有数据”。
- 出现会话异常、网络异常或后端错误时，用户反馈不清晰。

目标：

- 资源层明确区分 `loading`、`success`、`error`。
- 登录态、用户列表、链接加载都要能给出真实反馈。

#### 10. 修正复制公共链接的成功提示

当前复制功能没有等待浏览器剪贴板 Promise 成功就直接提示“已复制”。

目标：

- 只在剪贴板写入成功时提示成功。
- 权限不足或浏览器拒绝时展示错误反馈。

#### 11. 接入 diagnostics 与 cache 管理面板

后端已经具备以下接口，但当前前端未接入：

- `GET /api/users/{username}/diagnostics`
- `GET /api/users/{username}/cache`
- `POST /api/users/{username}/cache/refresh`
- `DELETE /api/users/{username}/cache`

目标：

- 在编辑器区域或侧栏中展示抓取诊断。
- 展示缓存状态、生成时间、过期时间、大小和行数。
- 支持手动刷新与清理缓存。

#### 12. 放宽账户更新流程

当前账户更新更接近“必须同时改密码”，不适合只改用户名的场景。

目标：

- 支持“仅改用户名”。
- 支持“仅改密码”。
- 仍然要求当前密码确认。

### P3: 质量与运维

#### 13. 补测试

当前仓库缺少系统化测试。

优先补齐：

- `packages/core` 的纯函数单元测试
- `auth` / `users` / `public` 路由集成测试
- cache 一致性测试
- SSRF / redirect / body limit 抓取链路测试

目标：

- 为认证、链接保存、公共聚合、缓存版本一致性建立回归保护。

#### 14. 增强观测信息

当前虽然有 tracing 和 fetch diagnostics，但整体仍偏“问题发生后人工排查”。

目标：

- 为限流命中、cache `hit/stale/miss/empty`、抓取失败原因增加更稳定的日志字段。
- 为关键请求补 request id 或可关联字段。
- 让公共路由行为更容易被日志与监控聚合。

#### 15. 文档与代码同步收口

当前阶段文档与前端现状存在轻微偏移，例如 diagnostics/cache 面板在文档中有描述，但当前 UI 并未实现。

目标：

- 功能补完后同步 README 和 `docs/rewrite/`。
- 区分“已完成阶段记录”和“后续计划”。

#### 16. 遗留代码清理

当前存在少量未实际使用的结构或辅助函数。

候选项：

- `frontend/src/components/console/view.rs`
- 未被使用的部分辅助 API

目标：

- 在行为稳定后做低风险清理。
- 避免无主代码继续增加理解成本。

## 分阶段实施计划

### 阶段 A: 安全与正确性收口

状态：已完成

已完成项：

- 密码处理改为保留原始输入，不再在认证和改密路径中对密码做 `trim()`
- 登录成功后轮换 session id，并重建登录态 CSRF token
- 启动期对关键配置项做显式校验，非法配置直接阻止服务启动
- 登录限流和公共路由限流响应补齐 `Retry-After`
- 前端复制公共入口链接改为等待剪贴板写入结果后再提示成功

目标：

- 修复最直接的认证与配置边界问题。

范围：

- 密码输入语义修正
- session / CSRF 轮换
- 启动期配置校验
- `Retry-After` 响应头
- 复制公共链接误报修正

完成标准：

- 认证流程不再依赖 `trim()` 密码。
- 登录后 session 身份发生轮换。
- 非法关键配置无法启动服务。
- 限流响应具备标准可消费头部。

### 阶段 B: 一致性与运行时内存边界

状态：已完成

已完成项：

- 新增数据库迁移，为 `fetch_diagnostics` 和 `user_cache_snapshots` 补外键与 `ON DELETE CASCADE`
- 启用 SQLite 连接级 `foreign_keys`
- `create_user`、`delete_user`、`set_links` 改为事务化写路径
- `delete_user` 改为依赖数据库级联清理，而不是应用层多段删除
- `set_links` 在单事务中完成用户配置更新与 cache / diagnostics 失效
- `DnsResolver` 与 `PinnedClientPool` 增加最大条目数限制和最旧项淘汰
- 链接安全校验改为受控并行，仍受 `CONCURRENT_LIMIT` 限制

目标：

- 降低脏数据和长时间运行内存增长风险。

范围：

- 用户写路径事务化
- 外键与级联删除
- DNS cache / pinned client pool 容量治理
- 评估或实现受控并行链接校验

完成标准：

- 删除用户和更新链接不再依赖多段非事务写入。
- 运行时缓存具备明确边界。

### 阶段 C: 前端运维能力补齐

目标：

- 让后端现有能力在 UI 中可见、可操作、可排错。

范围：

- diagnostics 面板
- cache 面板
- 资源状态错误显示
- 账户更新流程优化

完成标准：

- 管理台能查看抓取失败详情。
- 管理台能查看并控制 merged cache 生命周期。
- 加载失败和空状态在 UI 上可区分。

### 阶段 D: 测试、观测与文档收口

目标：

- 为后续迭代建立稳定回归面和可维护性基础。

范围：

- 单元测试与集成测试
- 关键日志字段增强
- README / 阶段文档同步
- 遗留代码清理

完成标准：

- 核心行为有回归测试覆盖。
- 关键运行路径更容易观察与排查。
- 文档重新与当前实现对齐。

## 建议实施顺序

建议严格按以下顺序推进：

1. 阶段 A
2. 阶段 B
3. 阶段 C
4. 阶段 D

原因：

- 阶段 A 改动回报最高，且能先收紧认证与运行边界。
- 阶段 B 会触及数据库和运行时结构，应该在安全收口后推进。
- 阶段 C 主要提升可见性，适合建立在稳定后端基础之上。
- 阶段 D 负责把前面改动沉淀为长期可维护资产。

## 明确暂不优先的事项

以下事项目前不列为最优先推进项：

- 将 `users.links` 从 JSON 文本改造成独立关系表
- 引入多管理员或 RBAC
- 引入任务队列、异步抓取调度器
- 做大规模前端视觉重构
- 改造成多服务拆分部署

原因：

- 当前瓶颈不在这里。
- 上述事项的收益低于认证正确性、一致性、可见性和测试缺口。
