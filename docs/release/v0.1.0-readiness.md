## 阶段二结果

完成日期：

- 2026-03-23

### 1. 安全告警结论

仓库级 Dependabot 告警检查命令：

```bash
gh api -H 'Accept: application/vnd.github+json' \
  '/repos/vansour/Submora/dependabot/alerts?state=open'
```

截至 2026-03-23，结果为空。

结论：

- 当前仓库没有 open Dependabot alerts
- 先前出现过的 `aws-lc-sys`、`quinn-proto`、`rustls-webpki` 等告警已处于 `fixed` 状态
- 使用 `trivy 0.69.3` 对 `ghcr.io/vansour/submora:v0.1.0-rc.4` 扫描 `HIGH/CRITICAL` 且 `--ignore-unfixed` 后，结果为 `0`

说明：

- 这里的结论只覆盖 GitHub 仓库依赖告警
- 镜像层面已经做过一轮高危复核，但正式版 tag 前仍建议再跑一次同口径扫描，避免基础镜像在发布窗口内发生变化

### 2. 升级与回滚边界

已验证升级路径：

- `v0.1.0-rc.3 -> v0.1.0-rc.4`

验证方式：

- 先用 `v0.1.0-rc.3` 镜像创建真实 SQLite 数据
- 再用 `v0.1.0-rc.4` 镜像启动同一数据目录
- 验证 migration、管理登录、diagnostics/cache 面板、公共路由 cache `hit`

明确边界：

- 一旦新版本启动时写入了新的 migration，旧版本二进制不保证还能直接读取该数据目录
- 以当前路径为例，数据库进入 migration `4` 后，`v0.1.0-rc.3` 会在启动阶段因 `VersionMissing(4)` 失败

阶段二要求的正式版口径：

- 升级前必须先备份 `data/`
- 回滚方案必须以“恢复升级前备份”为主
- 不应在 release note 中暗示“只回退镜像 tag 就能安全回滚”

### 3. 生产部署建议

正式版文档至少应持续保留以下建议：

- 公网部署前替换默认管理员密码
- HTTPS 反向代理后启用 `COOKIE_SECURE=true`
- 仅在可信代理完全接管入口流量时开启 `TRUST_PROXY_HEADERS=true`
- `data/` 纳入定时备份
- 以单机 SQLite 自托管为默认部署模型

### 4. 应写入 release note 的已知行为

- 当前只支持单管理员模型，没有 RBAC
- 当前没有后台定时抓取调度器
- 控制台外部访问公共路由后，已打开编辑器中的 diagnostics / cache 面板不会自动刷新
- 产品当前主要面向单节点 SQLite 部署，不是多副本共享数据库架构
