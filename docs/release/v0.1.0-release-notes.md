# v0.1.0 Release Notes Draft

本文档是 `v0.1.0` 正式版 release note 草案，面向最终 GitHub Release 文案整理。

## 概览

`v0.1.0` 是 `Submora` 当前重写阶段的首个正式版本。

这一版的重点不是继续扩张功能面，而是把已经落地的能力收口成一个可部署、可升级、可回归验证的单机自托管版本：

- 一个可直接使用的 Dioxus 管理台
- 一个面向订阅组的 Axum 管理 API 与公共聚合路由
- 一组围绕安全、可观测性、缓存一致性和发布质量的基础能力

## 核心能力

### 1. 管理台完成从壳层到可操作控制台的收口

- 管理台现在是一个单页工作台
- 支持创建、删除、排序订阅组
- 订阅链接支持逐行编辑、去重归一化、拖拽排序和公共入口复制
- 管理员账户支持仅改用户名或仅改密码，并要求当前密码确认
- 管理员账户更新后会立即使当前会话失效并要求重新登录

### 2. 运行状态面板可直接排查抓取问题

- 订阅组编辑区内置 diagnostics 面板
- 可查看每个上游链接的抓取状态、HTTP 状态、内容类型、大小、重定向次数与抓取时间
- `blocked` 抓取会在 UI 中直接显示为“已阻断”，不再落到模糊的未知状态

### 3. merged cache 进入持久化运行模型

- 公共聚合结果会写入持久 cache snapshot
- 管理台可查看 cache 状态、大小、行数、生成时间和过期时间
- 支持手动刷新缓存与清空缓存
- 公共路由响应继续通过 `x-substore-cache` 暴露 `hit` / `miss` / `stale` / `empty`

### 4. 安全边界比早期版本更明确

- 管理写接口继续要求 CSRF
- 登录成功后会轮换 session 标识与 CSRF token
- SSRF 校验、手动重定向复检、响应体体积限制都已接入
- 登录限流与公共路由限流现在都有标准 `Retry-After`
- 仅在 `TRUST_PROXY_HEADERS=true` 时才信任代理头部

### 5. 质量与发布资产已具备基本闭环

- `packages/core` 已有纯函数测试
- `backend/tests/integration_flows.rs` 已覆盖认证、公共路由、cache、diagnostics、SSRF/redirect/body limit 等关键链路
- `make release-check` 已作为发布前统一校验入口
- GitHub Actions 已支持 preview / prerelease / stable release 流程

## 升级说明

### 已验证升级路径

- `v0.1.0-rc.3 -> v0.1.0-rc.4`
- `v0.1.0-rc.4 -> v0.1.0`

### 升级前建议

```bash
docker compose down
cp -a data "data.backup.$(date +%Y%m%d-%H%M%S)"
docker compose pull
docker compose up -d
```

### 升级后最少检查项

- `GET /healthz`
- 管理员登录
- 至少一个已有订阅组的公共路由
- diagnostics / cache 面板仍可读取旧数据

### 回滚边界

- 数据库 migration 应按前向升级对待
- 一旦新版本把数据库迁移到新版本，旧二进制不保证还能直接读取该数据目录
- 以当前已验证路径为例，数据库进入 migration `4` 后，`v0.1.0-rc.3` 会因 `VersionMissing(4)` 启动失败

结论：

- 回滚方式应以“恢复升级前的 `data/` 备份”为主
- 不应把“只回退镜像 tag”当成正式支持的回滚方案

## 部署建议

- 公网部署前先替换默认管理员密码
- 建议放在 HTTPS 反向代理之后，并设置 `COOKIE_SECURE=true`
- 只有在可信代理完全托管入口流量时，才开启 `TRUST_PROXY_HEADERS=true`
- 当前默认部署模型是单机 SQLite，自托管边界就是 `data/`
- 请把 `data/` 纳入定时备份

## 已知限制

- 当前只支持单管理员模型，没有多管理员或 RBAC
- 当前没有后台定时抓取调度器
- 已打开的 diagnostics / cache 面板不会在控制台外部公共访问后自动刷新
- 当前主要面向单节点 SQLite 部署，不是多副本共享数据库架构

## 兼容性与破坏性说明

- 相比 `rc.4`，`v0.1.0` 不计划引入新的 HTTP API 破坏性变更
- 对外语义仍保持：
  - 管理台入口 `/`
  - 公共聚合路由 `GET /{username}`
  - 既有 `/api/auth/*` 与 `/api/users/*` 管理接口
- 运行时最重要的兼容性边界在数据库 migration，而不是接口路径
